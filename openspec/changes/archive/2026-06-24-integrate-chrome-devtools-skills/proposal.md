# Proposal

## Problem

This repository already exposes a `chrome-devtools` MCP server, but the Chrome DevTools upstream project now publishes a fuller bundle: MCP server, CLI, skills, and harness-specific plugin or extension entry points. The current repo state does not track the upstream skills, does not distinguish plugin-managed harnesses from repo-managed MCP fallback harnesses, and does not enforce the invariant that each harness has exactly one active Chrome DevTools MCP source.

Without a tracked integration, future syncs can duplicate MCP entries, omit upstream skills from harnesses that rely on repo skill projection, lose provenance for imported external content, or enable telemetry/performance network behavior without explicit policy.

## Intent

Integrate `ChromeDevTools/chrome-devtools-mcp` as a three-part capability across supported AI tools: upstream plugin or extension where supported, repo-adapted skills for portable skill projection, and the existing canonical MCP registry as the fallback for MCP-only clients.

## Scope

- Create a tracked Chrome DevTools upstream intake record for commit `a90378adf3226e8b27a05cdcfdd801c199acaa93`, package `chrome-devtools-mcp` version `0.23.0`, license `Apache-2.0`, author `Google LLC`.
- Promote the six upstream skills into repo `skills/` with repo-safe names, valid frontmatter, provenance notes, safe-output guidance, and no hidden hooks or automatic installs.
- Preserve repo names `chrome-devtools` and `chrome-devtools-cli`; prefix the broad upstream skills as `chrome-devtools-a11y-debugging`, `chrome-devtools-debug-optimize-lcp`, `chrome-devtools-memory-leak-debugging`, and `chrome-devtools-troubleshooting`.
- Track per-harness Chrome DevTools source ownership as `plugin`, `extension`, `repo-mcp`, `manual-ui`, or `blind-spot` in the plugin/extension registry.
- Harden the canonical MCP launch policy for usage statistics, update checks, and optional CrUX behavior while preserving the existing headed persistent-profile default and the OpenCode local wrapper override.
- Add dedupe and validation behavior so plugin-managed harnesses do not also receive an active standalone repo MCP entry for `chrome-devtools`.
- Update generated catalogs and docs through repo generators after skill changes.

## Out Of Scope

- Editing generated downstream surfaces directly, including `.github/copilot-instructions.md`, `.github/instructions/**`, `.vscode/mcp.json`, `mcp.json`, generated `opencode.json`, or generated OpenSpec tool artifacts.
- Enabling unsupported plugin integrations for Cursor, Codex, OpenCode, Antigravity, Claude Desktop, ChatGPT, Perplexity Desktop, or Cherry Studio without verified plugin support.
- Changing the OpenCode wrapper behavior documented in `instructions/opencode-global.md` except to preserve it as the OpenCode-specific override.
- Committing trace files, screenshots, Lighthouse reports, heap snapshots, browser profiles, or other runtime artifacts generated by Chrome DevTools workflows.

## Affected Users And Tools

- Repo maintainers who manage skills, MCP registry entries, plugin manifests, and downstream harness sync.
- Claude Code users who should install the upstream Chrome DevTools plugin.
- Gemini CLI users who should install the upstream Chrome DevTools extension.
- VS Code and GitHub Copilot users who can consume the upstream plugin from source where supported.
- Codex, OpenCode, Cursor, Claude Desktop, ChatGPT, Perplexity Desktop, Cherry Studio, and Antigravity users that depend on repo MCP config, repo skill projection, presets, or documented manual UI setup.

## Generated Surfaces To Refresh

- `README.md` via `uv run wagents readme` if the skill catalog changes.
- Docs site content via `docs-steward` and repo docs generators after skills are promoted.
- Harness import bundles via existing sync scripts only after source-of-truth files are updated.

## Risks

- Plugin-managed and repo-managed MCP entries can collide if dedupe rules are incomplete.
- Chrome DevTools performance tooling can send trace URLs to Google CrUX unless the integration policy explicitly disables or documents that behavior.
- Upstream CLI examples can encourage global installs or writing local artifacts; adapted skills must keep those actions explicit and safe.
- Memory and performance workflows can create sensitive local browser artifacts; skills must warn not to commit traces, profiles, screenshots, Lighthouse reports, or heap snapshots.
- The repo is currently dirty with unrelated user changes; implementation must avoid reverting, restaging, or modifying unrelated files.
