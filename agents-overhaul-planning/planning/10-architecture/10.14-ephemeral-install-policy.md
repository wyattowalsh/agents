# Ephemeral Install Policy

## Objective

Prefer repeatable, low-friction installation flows that avoid mutating global state unless the user explicitly requests it.

## Preferred installers

| Artifact | Preferred install | Reason | Caveat |
|---|---|---|---|
| Agent Skills | `npx skills add ...` or `gh skill install ...` | broad host support and lifecycle commands | verify content before install |
| Python MCP | `uvx <package>` | isolated execution, no project dependency pollution | pin package/ref for production |
| Node MCP | `npx -y <package>@<version>` | ephemeral package execution | `@latest` not allowed in validated profiles |
| Local MCP source | package and run via `uvx --from`/`npx github:` | avoids absolute local paths | requires packaging work |
| Native plugin | harness-specific marketplace/install command | best UX where stable | support varies by harness |

## Versioning rules

1. `@latest` is allowed in discovery/experimental profiles only.
2. Validated profiles must pin version, tag, or immutable SHA.
3. Update commands must produce before/after manifest diff.
4. Rollback must reinstall previous pinned version.
5. External skills require preview/audit before install.

## npx vs gh skill

Use `npx skills` as the cross-agent default. Use `gh skill` for Copilot-specific workflows where provenance metadata, preview, pinning, and publishing checks are valuable.

## uvx vs local paths

Absolute local paths are acceptable only for developer-private profiles. Portable repo profiles must use package names, Git refs, or documented local source build steps.
