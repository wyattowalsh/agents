# Registry Design

## Registry files

Proposed canonical files:

```text
config/registry/
  harnesses.yaml
  capabilities.yaml
  skills.yaml
  plugins.yaml
  mcps.yaml
  openspec.yaml
  support-tiers.yaml
  trust-tiers.yaml
```

## Design rules

- Split registries by type for ownership and merge safety.
- Use stable IDs, not display names.
- Store generated output paths separately from source paths.
- Include `source_url`, `source_kind`, `license`, `maintainer`, and `last_verified_at` for all external artifacts.
- Record `preferred_install_surface` and `fallback_install_surface`.
- Record why each MCP is not a skill.

## Generated outputs

- README skill/agent/harness tables
- docs site pages
- per-harness install snippets
- support tier matrix
- conformance fixtures
- OpenSpec capability index

## Acceptance criteria

- Schema validation exists for every registry.
- Registry updates fail CI unless docs are regenerated.
- External artifacts cannot become first-class without source and validation metadata.
