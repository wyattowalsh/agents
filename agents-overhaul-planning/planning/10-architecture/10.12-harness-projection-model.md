# Harness Projection Model

## Objective

Define how canonical repo capabilities become harness-specific artifacts without duplicating business logic.

## Canonical inputs

- `planning/manifests/harness-registry.yaml`
- `planning/manifests/skill-registry.schema.yaml`
- `planning/manifests/mcp-registry.schema.yaml`
- `agent-bundle.json`
- repo inventory snapshot
- selected environment overlays

## Projection outputs

| Canonical capability | Claude Code | Copilot | Cursor | Gemini CLI | OpenCode | ChatGPT/Codex | Cherry/Perplexity/Antigravity |
|---|---|---|---|---|---|---|---|
| Skill | `.claude/skills`, plugin `skills/` | `.github/.claude/.agents/skills`, `gh skill`/CLI | `.agents/skills` or compatible path | extension context or skill fallback | `.opencode/.claude/.agents/skills` | AGENTS.md/Codex/Apps proxy | native if supported; otherwise docs/instructions |
| Persistent instructions | `CLAUDE.md`, settings, rules | `.github/copilot-instructions.md`, `.github/instructions`, `AGENTS.md` | `.cursor/rules`, `AGENTS.md`, `CLAUDE.md` | `GEMINI.md`, extension context | `AGENTS.md`, opencode config | GPT instructions / AGENTS.md | harness-specific wrapper |
| MCP | `.mcp.json`, plugin MCP | repo MCP config, VS Code MCP, custom agent MCP | `mcp.json` global/project | `settings.json`, extension `mcpServers` | config MCP | Apps SDK / Codex config | UI config/helper |
| Agent profile | subagents | `.github/agents/*.md`, CLI agents | background/web agents | custom commands/extensions | agent configs | Codex tasks / GPTs | experimental |
| Plugin | Claude plugin bundle | Copilot extension/skillset/custom agent | VS Code extension adjacent | Gemini extension | OpenCode plugin | Apps SDK/Actions | native app presets |

## Projection principles

1. Canonical manifests remain source of truth.
2. Adapters render candidate output to a staging area first.
3. Transaction engine owns writes into repo or user config.
4. Projection tests compare generated output against golden fixtures.
5. Harness-specific docs must name unsupported fields and preview/beta behavior.

## Failure modes

- Same skill projected into multiple directories with divergent contents.
- Harness-specific instructions drift from `instructions/global.md`.
- MCP tools silently differ between CLI/editor/cloud surfaces.
- Plugins pin or unpin versions inconsistently.
- Generated README claims support that registry marks experimental.

## Acceptance criteria

- Every adapter has a fixture for at least one skill, one instruction file, and one MCP candidate.
- Projection output includes source manifest references and generated-file comments.
- Unsupported harness fields are dropped with warnings, not silently.
