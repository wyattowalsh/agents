# OpenSpec Governance Integration

## Objective

Use OpenSpec to govern major behavior and architecture changes, especially changes that affect generated configs, harness compatibility, skill packaging, security policy, or CLI UX.

## OpenSpec roles in this repo

- Capture user-visible capability changes before implementation.
- Keep proposals, design notes, and task lists together.
- Define spec deltas for the agent control plane, skills-first extension model, harness registry, MCP live-systems layer, config transaction model, documentation truth model, and UI/UX automation.
- Provide a changelog-like history once changes are archived.

## Required commands / CLI integration

The `wagents openspec` command group should eventually support:

- `wagents openspec status`
- `wagents openspec new <change-id>`
- `wagents openspec validate`
- `wagents openspec graph`
- `wagents openspec sync-planning`
- `wagents openspec archive <change-id>`

## Reconciliation requirement

Because OpenSpec assets may already exist in the repo, implementation must begin by inventorying and preserving existing files. Generated bundle files must not overwrite repo OpenSpec assets without a diff review.

## Acceptance criteria

- P0/P1 task graph nodes map to OpenSpec tasks.
- Specs contain acceptance criteria that CI can validate where possible.
- Planning docs and OpenSpec files do not contradict one another.
- Archive flow is documented before implementation completes.
