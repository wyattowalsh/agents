# Extension Decision Tree: Skill vs Plugin vs MCP vs OpenAPI vs Instruction

## Objective

Prevent extension sprawl by choosing the simplest portable extension surface that satisfies the capability.

## Decision tree

```text
Is the capability mostly static knowledge, repeatable workflow, or local script?
  yes -> Agent Skill
  no  -> continue

Does it require live external state, authenticated SaaS, browser runtime, DB/search/vector state, or telemetry?
  yes -> MCP or API adapter
  no  -> continue

Is it harness-specific UI/lifecycle behavior that cannot be represented as a skill?
  yes -> Harness plugin/extension
  no  -> continue

Is it a REST API action for ChatGPT/custom GPT-style usage?
  yes -> OpenAPI / GPT Action adapter
  no  -> continue

Is it an always-on coding convention or repo policy?
  yes -> AI instruction/rules file
  no  -> skill or docs-only reference
```

## Skill is preferred when

- It can be packaged with `SKILL.md` and optional scripts/references/assets.
- It can be invoked locally or by the model without a long-running service.
- It is reusable across at least two harnesses.
- It has deterministic CLI behavior or static reference material.
- It can be validated in CI.

## MCP is preferred when

- The agent needs real-time browser/runtime state.
- The agent needs authenticated external API state.
- The operation is inherently interactive or long-lived.
- The capability needs dynamic tool discovery from a remote system.
- The protocol's resource/prompt/tool lifecycle materially improves the UX.

## Plugin/extension is preferred when

- The harness offers native lifecycle hooks that cannot be represented as skills.
- The capability needs to register commands/menus/UI/agents inside the harness.
- The plugin can bundle skills while providing harness-native install/update semantics.

## OpenAPI/function calling is preferred when

- The target surface is ChatGPT Actions or another REST API tool model.
- The capability already has a stable HTTP API.
- Authentication and privacy-policy requirements can be satisfied.

## Instruction/rules files are preferred when

- The behavior is always-on policy.
- The content is short and stable.
- The instruction must affect code generation globally or by path.

## Anti-patterns

- Using MCP to deliver static prompt packs.
- Installing remote MCP servers for tasks that a pinned CLI skill can do locally.
- Treating all harness-specific rules as equivalent.
- Encoding secrets in generated config files.
- Making preview APIs look stable in docs.
