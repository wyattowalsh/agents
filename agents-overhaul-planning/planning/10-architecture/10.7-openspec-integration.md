# OpenSpec Integration Model

## Objective

Use OpenSpec as the durable change-governance layer for the overhaul.

## Why OpenSpec fits

OpenSpec is a lightweight spec-driven development workflow where specs live in the repository, changes produce proposal/design/tasks/spec deltas, and archived changes consolidate knowledge into durable specs. It supports many AI coding assistants and is designed for brownfield projects.

## Proposed directory usage

```text
openspec/
  specs/
    agent-control-plane/spec.md
    skills-first-extension-model/spec.md
    harness-registry/spec.md
    mcp-live-systems-layer/spec.md
    config-transaction-model/spec.md
    ui-ux-automation/spec.md
  changes/
    agents-platform-overhaul/
      proposal.md
      design.md
      tasks.md
      specs/
        agent-control-plane/spec.md
        skills-first-extension-model/spec.md
        harness-registry/spec.md
        mcp-live-systems-layer/spec.md
        config-transaction-model/spec.md
        ui-ux-automation/spec.md
```

## Command support

Add `wagents openspec` commands:

```text
wagents openspec status
wagents openspec new <change-id>
wagents openspec validate
wagents openspec graph
wagents openspec sync-planning
wagents openspec archive <change-id>
```

## Integration rules

- Every P0/P1 work package must map to an OpenSpec task.
- Every new stable capability must have or update an OpenSpec requirement.
- `planning/` may contain broad program detail; `openspec/specs/` captures stable capability truth.
- OpenSpec artifacts should be checked into git and reviewed before code changes.

## Existing-content caution

Because you indicated OpenSpec assets already exist in the repo, the first implementation step must inspect and preserve those assets. Do not overwrite them.

## Acceptance criteria

- OpenSpec change folder exists for the overhaul.
- OpenSpec specs capture capability-level requirements.
- CI validates OpenSpec paths and task completion consistency.
- Docs link from planning docs to OpenSpec specs.
