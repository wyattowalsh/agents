# OpenSpec Reconciliation Protocol

## Objective

Avoid overwriting existing OpenSpec content while making OpenSpec the planning governance layer.

## Protocol

1. Inventory current `openspec/` tree.
2. Classify permanent specs vs active changes.
3. Detect existing change IDs and tasks.
4. Map planning task graph clusters to active or proposed OpenSpec changes.
5. Add missing spec deltas only after preserving existing files.
6. Validate OpenSpec before implementation.
7. Archive completed changes into permanent specs.

## Active overhaul change shape

```text
openspec/changes/agents-platform-overhaul/
  proposal.md
  design.md
  tasks.md
  specs/
    agent-control-plane/spec.md
    skills-first-extension-model/spec.md
    mcp-live-systems-layer/spec.md
    harness-registry/spec.md
    config-transaction-model/spec.md
    documentation-truth/spec.md
```

## Acceptance criteria

- `wagents openspec doctor` reports no orphaned task IDs.
- Every P0/P1 task graph node maps to an OpenSpec task or explicit exception.
- OpenSpec specs do not duplicate planning docs; they define requirements.
