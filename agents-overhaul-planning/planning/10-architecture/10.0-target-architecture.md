# Target Architecture

## Objective

Define a single architecture that can represent skills, MCPs, plugins, instructions, harness-specific config, and OpenSpec changes without duplicating logic across harnesses.

## Layers

```text
┌────────────────────────────────────────────────────────────┐
│ UX Layer                                                    │
│ wagents doctor | catalog | install-skill | sync | rollback │
├────────────────────────────────────────────────────────────┤
│ Orchestration Layer                                         │
│ repo inventory | registry resolver | transaction engine     │
├────────────────────────────────────────────────────────────┤
│ Canonical Registry Layer                                    │
│ skills | MCPs | plugins | harnesses | docs | OpenSpec       │
├────────────────────────────────────────────────────────────┤
│ Adapter Layer                                               │
│ skills | plugins | MCP | OpenAPI | CLI | rules | OpenSpec   │
├────────────────────────────────────────────────────────────┤
│ Harness Projection Layer                                    │
│ Claude | Copilot | OpenCode | Gemini | Cursor | etc.        │
├────────────────────────────────────────────────────────────┤
│ Validation Layer                                            │
│ schema | golden fixtures | smoke tests | docs truth | evals │
└────────────────────────────────────────────────────────────┘
```

## Core interfaces

### RegistryResolver

Inputs:

- canonical manifests;
- repo inventory;
- environment overlay;
- selected harnesses;
- user policy.

Outputs:

- desired state;
- support-tier claims;
- generated docs data;
- validation plan.

### Adapter

Inputs:

- desired state;
- harness capability model;
- adapter-specific constraints.

Outputs:

- file operations;
- CLI operations;
- test operations;
- rollback metadata.

### TransactionEngine

Inputs:

- planned operations;
- current filesystem/config state;
- safety policy.

Outputs:

- preview diff;
- applied transaction;
- rollback snapshot;
- audit log.

### ConformanceRunner

Inputs:

- manifests;
- golden fixtures;
- generated output;
- harness smoke commands.

Outputs:

- pass/fail report;
- drift report;
- remediation hints.

## Non-goals

- Do not make every harness support every capability.
- Do not hide preview/unstable surfaces behind stable labels.
- Do not install or execute third-party tools without explicit policy.
- Do not trust MCP directories as safety attestations.

## Acceptance criteria

- Every generated config is traceable to a registry record.
- Every external capability has a source, support tier, and security classification.
- Every harness projection has golden fixtures.
- The transaction engine can roll back failed writes.
