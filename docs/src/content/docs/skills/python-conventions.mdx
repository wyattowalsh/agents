## What It Does

This skill enforces a consistent Python tooling stack across the project: [`uv`](https://docs.astral.sh/uv/) for package and environment operations, [`ty`](https://docs.astral.sh/ty/) for type checks, [`ruff`](https://docs.astral.sh/ruff/) for linting and formatting, and [`pytest`](https://docs.pytest.org/) for tests. It keeps detailed library, testing, performance, and exception guidance behind reference files so the active skill stays lightweight.

<Aside type="note">
This is an **auto-invoke** skill ŌĆö it activates automatically when the agent detects you're working on `.py` files or `pyproject.toml`. It is not visible in the `/` slash command menu.
</Aside>

## Tooling

- **Package manager**: [`uv`](https://docs.astral.sh/uv/) for all Python operations (`uv run python ...` instead of `python`, `uv add` instead of `pip install`)
- **Dependencies**: `uv add`, `uv add --group dev`, `uv remove`, and `uv lock --upgrade-package <pkg>` as appropriate
- **Reproducible installs**: `uv sync --locked` for lockfile-backed validation
- **Project config**: `pyproject.toml` with `uv` dependency metadata (never `uv pip install` or `pip install` as the default path)
- **Type checking**: `uv run ty check` ŌĆö using [`ty`](https://docs.astral.sh/ty/) (not `mypy`)
- **Linting & formatting**: `uv run ruff check` and `uv run ruff format` ŌĆö using [`ruff`](https://docs.astral.sh/ruff/)
- **Testing**: `uv run pytest` ŌĆö using [`pytest`](https://docs.pytest.org/)
- **Task running**: `uv run <command>` for all script execution

### Virtual Environments

- Let `uv` manage virtual environments automatically
- Never manually create or activate `.venv` directories
- Use `uv run` to execute within the project environment

## Preferred Libraries

When starting new Python work and no stronger local constraint exists, use `references/library-preferences.md` for guided defaults. These choices are preferences, not mandatory rewrites:

| Numerics | `numpy` | manual math |
| MCP servers | `fastmcp` | raw MCP protocol |

## Project Structure

- Use `pyproject.toml` for all project metadata and dependencies
- Place source code in a package directory matching the project name
- Use `uv` workspace members for monorepo sub-packages

## Critical Rules

<Aside type="caution" title="Non-negotiable">
Rules 1ŌĆō3 are hard requirements. Breaking them causes CI failures and inconsistent environments. Check `references/exceptions.md` before deviating from any convention.
</Aside>

1. Reject `pip install`, `uv pip install`, or bare `python` unless an approved exception applies
2. Use `uv add` for runtime dependencies and `uv add --group dev` for dev-only dependencies
3. Use `uv run ty check` for type checking; do not recommend `mypy` or bare `ty check` as the default
4. Run `uv run ruff check`, `uv run ruff format`, `uv run ty check`, and `uv run pytest` as the default completion gate
5. Do not edit `uv.lock` by hand
6. Keep guided library preferences optional when a local stack already exists
7. Check `references/exceptions.md` before breaking any convention

**Canonical terms** (use these exactly):
- `uv` ŌĆö the required package manager and task runner
- `ty` ŌĆö the required type checker (not mypy)
- `uv run ty check` ŌĆö the required type-check command
- `ruff` ŌĆö the required linter and formatter
- `pyproject.toml` ŌĆö the single source of project configuration
- `uv run` ŌĆö prefix for all Python command execution
- `uv sync --locked` ŌĆö default reproducible install check for lockfile-backed workflows

## Critical Rules

1. Reject Python setup advice that uses `pip install`, `uv pip install`, or bare `python` when the task is not explicitly on an approved exception path.
2. Require `uv add` for runtime dependencies and `uv add --group dev` for development-only dependencies unless `references/exceptions.md` justifies a legacy or constrained-environment deviation.
3. Require `uv run ty check` for type checking; do not recommend `mypy` or bare `ty check` as the default path in this repo.
4. Treat `uv run ruff check`, `uv run ruff format`, `uv run ty check`, and `uv run pytest` as the default completion gate for Python changes unless an exception is documented.
5. Do not edit `uv.lock` by hand; use `uv lock`, `uv sync`, or dependency commands.
6. Do not present guided library preferences as mandatory replacements for an already-established local stack.
7. Read `references/exceptions.md` before approving a legacy toolchain, alternate environment manager, or alternate library path.
8. Redirect mixed-language or non-Python-primary work through `references/redirection-boundaries.md` instead of force-fitting this skill onto the whole task.
