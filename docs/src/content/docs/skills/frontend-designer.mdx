## What It Does

The frontend-designer skill encodes modern frontend best practices into a structured, multi-mode workflow. Instead of relying on general knowledge that may be outdated, it provides **research-backed references** for each technology in the stack and enforces **12 critical rules** that prevent common mistakes.

## Usage Examples

<Tabs>
  <TabItem label="Scaffold">
    ```
    /frontend-designer scaffold my-dashboard
    ```
    Creates a new project with Tailwind v4 CSS-first config, shadcn/ui, distinctive typography, and the recommended file structure. Defaults to Vite + React unless you specify Next.js or Remix.
  </TabItem>
  <TabItem label="Component">
    ```
    /frontend-designer component pricing-card
    ```
    Designs and builds a single component. Classifies as server or client based on your framework, gathers design intent, selects Radix primitives, and implements with container queries and accessibility.

    Also works with natural language:
    ```
    /frontend-designer build a responsive sidebar with collapsible navigation
    ```
  </TabItem>
  <TabItem label="Theme">
    ```
    /frontend-designer theme
    ```
    Configures your project's design token system: primitive → semantic → component token layers in `@theme {}` with oklch colors and fluid type scale.

    Extract tokens from existing hardcoded values:
    ```
    /frontend-designer tokens
    ```
  </TabItem>
  <TabItem label="Refactor">
    ```
    /frontend-designer refactor src/components/
    ```
    Scans for legacy patterns (Tailwind v3 config, media queries, hardcoded colors, excessive `"use client"`), presents a migration plan, and executes after approval.
  </TabItem>
  <TabItem label="Audit">
    ```
    /frontend-designer audit
    ```
    Read-only analysis producing a severity-ranked report across 6 dimensions. Recommends actions linked to other modes (e.g., "run `/frontend-designer refactor`").
  </TabItem>
</Tabs>

## Critical Rules

12 rules enforced across all modes — violations are flagged automatically:

1. **Never use `tailwind.config.js`** — Tailwind v4 is CSS-first with `@import "tailwindcss"` and `@theme {}`
2. **Framework-aware components** — Next.js/Remix: minimize `"use client"`. Vite+React: `"use client"` is unnecessary
3. **Radix primitives via shadcn/ui** — never build custom dialogs, popovers, or dropdowns from scratch
4. **Design tokens in `@theme {}`** — never hardcode colors, spacing, or font values
5. **Container queries for components** — media queries only for page layout and user preferences
6. **`@supports` guards** — mandatory for `contrast-color()`, anchor positioning, scroll-driven animations, `@scope`
7. **Logical properties** — `margin-inline-start` not `margin-left` for RTL readiness
8. **Distinctive fonts** — never default to Inter, Roboto, Arial, or system-ui; Monaspace for code
9. **Accessibility non-negotiable** — keyboard nav, ARIA, focus management, WCAG AA contrast
10. **Live docs on demand** — consult Context7/WebSearch only when references may be outdated
11. **Body under 500 lines** — deep details live in 8 reference files
12. **Audit is read-only** — never modify code during audit mode
