## What It Does

Search for job opportunities, score them against a user's career profile, and populate the Ronin pipeline for human approval.

## Quick Start

```
1. mcp__ronin__get_preferences(user_id="<user-id>")         → get target_roles, industries, geographic_preferences
2. mcp__brave-search__brave_web_search(query="...", count=10) → search for jobs (multiple queries)
3. mcp__ronin__create_opportunity(user_id, data={...})    → save each result (auto-scores on creation)
4. mcp__ronin__compute_match_score(user_id, opportunity_id) → re-score if needed
5. Present results grouped by score tier → user approves (→ shortlisted) or rejects (→ withdrawn)
```

## Critical Rules

1. Always check existing pipeline URLs for deduplication before creating new opportunities — call `list_opportunities` at startup
2. Must load preferences before running any search — an empty `target_roles` list means discovery has no search intent
3. Always apply agent judgment beyond raw match scores — lexical overlap misses seniority, location, and industry fit
4. Never auto-approve or auto-reject without explicit user confirmation — present results for human review
5. Must deduplicate by URL before calling `create_opportunity` — duplicate entries pollute the pipeline
6. Always surface reject/approve patterns to the user after a review session to inform preference refinement
7. Never present opportunities with `match_score &lt; 30` unless the user explicitly asks for them
8. Must set the `source` field on every created opportunity to the channel that found it (e.g., "brave_search", "tavily", "career_page", "domain_board")
9. Always fetch full page content for short search snippets before creating opportunities — longer descriptions produce better scores
10. Never reference Voyage AI semantic embeddings or multi-dimensional sub-scores as current features — they are roadmap items

## Overview

This skill orchestrates the **job discovery loop** — the primary mechanism by which Ronin surfaces relevant opportunities. It feeds every downstream workflow (materials generation, application tracking, interview prep).

### Runtime contract

Treat this skill as an **on-demand sourcing workflow** driven by live MCP tools, not as a promise about a specific background scheduler or agent runtime. The reliable contract is:

```
load context → run external searches → deduplicate/filter → create opportunities → present for human review
```

- **Context** — `get_preferences`, `get_user_memory`, and `list_opportunities` provide search intent, personalization hints, and dedupe context.
- **Sourcing** — External search/fetch tools provide candidate listings and company-career-page detail.
- **Persistence** — `create_opportunity` writes opportunities in the `discovered` state and creates the linked pipeline entry.
- **Review** — `approve_opportunity` moves an opportunity to `shortlisted`; `reject_opportunity` records a user rejection as `withdrawn`.

You may use a 4-channel search strategy, but do not claim any fixed scheduler, background cadence, or auto-written discovery memory unless the current product surface explicitly provides it.

### State-management boundary

This skill only documents the live discovery MCP surface for sourcing plus explicit human review. It does not implement broader pipeline-state orchestration, hidden schedulers, or background state-management beyond `create_opportunity`, `approve_opportunity`, and `reject_opportunity`.

### Request router

Route the request before loading extra references:

- **Configure** — view or update preferences before discovery runs.
- **Run discovery** — source new opportunities from search and career-page inputs.
- **Review** — inspect existing discovered opportunities and decide approve/reject.
- **Health / funnel** — inspect funnel health and conversion bottlenecks.

If the request mixes sourcing and review, run sourcing first, present a capped review batch, then handle explicit approve/reject actions. Do not auto-approve or auto-reject.

### Scoring methodology

Two methods exist: automated graph (Jaccard similarity) for batch runs, and MCP tool (cosine similarity on hash vectors) for on-demand. Both are **lexical overlap** — they measure shared vocabulary, not semantic meaning. See `references/data-shapes.md` for details. Apply agent judgment (Step 5) to compensate.

### Scaling strategy

Keep discovery runs bounded and reviewable:

- Run first-pass search and fetch channels in parallel.
- Deduplicate by canonical job URL before any write calls.
- Cap the first review batch instead of dumping every result at once.
- If unique high-signal results are thin, broaden queries incrementally; if noisy, tighten around company, role, and location cues.
- Prefer official careers pages and fetched page content when search-engine snippets are weak, stale, or too thin for meaningful lexical scoring.

## Prerequisites

1. **Populated Ronin profile** — Run `ronin-profile-builder` first. An empty profile produces meaningless scores.
2. **Configured preferences** — Set at minimum `target_roles` and either `geographic_preferences` or `remote_preference: "remote"`. See Configure Preferences below.
3. **`BRAVE_API_KEY`** — Required for `brave_web_search`. Set in environment.
4. **Ronin MCP tools available** — In product use this comes from the connected Ronin MCP server. For local development, run the API app mounted at `/mcp`.
5. **Valid `user_id`** — Ronin MCP signatures keep `user_id: str` at the boundary. Pass the active Ronin user identifier exactly as supplied by the current environment. In the current local implementation, some discovery-adjacent MCP paths still validate UUID-shaped values, so treat format mismatches as runtime constraints rather than rewriting IDs manually.

## Workflow: Configure Preferences

Always inspect existing preferences before overwriting. Preferences persist across sessions.

### Step 1 — Read current preferences

```python
mcp__ronin__get_preferences(user_id="<user-id>")
```

Returns the full preferences dict including `target_roles`, `industries`, `deal_breakers`, `remote_preference`, `compensation_goals`, and `geographic_preferences`. If no preferences exist, `preferences` will be `null`. An empty `target_roles` list means discovery has no search intent — populate it before running. See `references/tool-reference.md` for the full response shape.

### Step 2 — Update preferences

Only the keys you provide are updated — omitted keys retain their current values.

```python
mcp__ronin__update_preferences(
    user_id="<user-id>",
    data={
        "target_roles": ["Staff Engineer", "Principal Engineer"],
        "industries": ["fintech", "devtools", "AI/ML"],
        "deal_breakers": ["no_equity", "on_call_24_7", "contract_only"],
        "remote_preference": "remote"
    }
)
```

Updatable fields: `target_roles`, `target_companies`, `company_sizes`, `industries`, `deal_breakers`, `remote_preference`. Note: `compensation_goals` and `geographic_preferences` require direct profile editing. See `references/tool-reference.md` for the full field table.

## Workflow: Run Discovery

Run this after preferences are configured and the profile is populated.

### Step 1 — Self-fetch context

Load context in parallel:

```python
prefs = mcp__ronin__get_preferences(user_id="<user-id>")
memory = mcp__ronin__get_user_memory(user_id="<user-id>")
existing = mcp__ronin__list_opportunities(user_id="<user-id>", state="discovered")
```

Extract: `target_roles`, `industries`, `geographic_preferences`, `deal_breakers` from prefs; personalization signals and `reject_reasons` from memory; existing URLs for deduplication.

### Step 2 — Build and execute multi-channel search

Use the 4-channel sourcing strategy. Emit all search tool calls in parallel:

1. **Standard boards** — Brave (ATS-scoped) + Tavily (open industry)
2. **Funding signal detection** — Tavily research on recently funded companies
3. **Company career pages** — `fetch_markdown`/`fetch_html` on target company URLs
4. **Domain-specific boards** — Fetch and parse niche job boards

> **DECISION POINT:** Issue all 4 channels in parallel on the first pass. If total unique results < 10, broaden queries. Adjust based on `reject_reasons` from prior sessions.

If general search results are noisy or snippet-only, pivot quickly to official careers pages or fetched job-post pages before writing anything.

See `references/tool-reference.md` for full tool call examples for each channel.

### Step 3 — Create each opportunity

Deduplicate by URL first. For each unique result, extract structured data and create:

```python
mcp__ronin__create_opportunity(
    user_id="<user-id>",
    data={
        "title": "Staff Software Engineer",
        "company": "Stripe",
        "url": "https://stripe.com/jobs/listing/3982734812",
        "description": "...",
        "required_skills": ["Python", "distributed systems"],
        "source": "brave_search"
    }
)
```

Required fields: `title`, `company`, `url`. The tool computes an initial match score at creation time. See `references/tool-reference.md` for the full field table and response shape.

> **Tip:** If the search snippet is short, fetch the full page content before calling `create_opportunity`. Longer descriptions produce better lexical overlap scores.

### Step 4 — Re-score if needed

`create_opportunity` auto-scores. Call `compute_match_score` to refresh after profile updates or for stale scores. Returns `match_score` (0-100) and `reasoning`.

### Step 5 — Apply agent judgment (re-ranking)

The lexical match score misses seniority alignment, industry fit, and location compatibility. Before presenting:

1. **Seniority check** — Does the title level match target roles?
2. **Location/remote fit** — Matches `geographic_preferences` and `remote_preference`?
3. **Industry alignment** — Company's industry in the user's `industries` list?
4. **Freshness** — Prefer recently posted roles

> **DECISION POINT:** If `match_score >= 50` but seniority is wrong, demote. If `match_score` is 30-50 but title and company exactly match preferences, promote. Use judgment beyond raw score.

### Step 6 — Filter and present

Present opportunities with `match_score >= 30` grouped by tier: strong (>= 60), moderate (30-59), weak (< 30, not shown by default). For each: title, company, location, remote status, match score/tier, URL, and your seniority/fit assessment. Offer batch operations: "Approve all strong matches?" or "Skip remaining?"

### Step 7 — Approve or reject

**Approve** — transitions to `shortlisted`:

```python
mcp__ronin__approve_opportunity(user_id="<user-id>", opportunity_id="opp-uuid")
```

**Reject** — transitions to `withdrawn`:

```python
mcp__ronin__reject_opportunity(user_id="<user-id>", opportunity_id="opp-uuid")
```

After a review session, summarize patterns: "You approved 3 fintech roles and rejected 4 devtools roles — should I adjust your industry preferences?"

## Workflow: Review Existing Opportunities

Use when the user wants to review opportunities already in the pipeline.

```python
mcp__ronin__list_opportunities(user_id="<user-id>", state="discovered", min_score=30, sort_by="match_score")
```

Results capped at 50. Filter by `min_score` or `state` to narrow. For full details on a single opportunity, call `get_opportunity(user_id, opportunity_id)`. Then follow the approve/reject flow from Step 7. See `references/tool-reference.md` for parameters and response shapes.

## Workflow: Pipeline Health Check

```python
mcp__ronin__get_funnel_stats(user_id="<user-id>")
```

Returns per-stage cumulative counts and conversion rates. A low `approved_per_discovered` rate suggests queries are too broad or thresholds need adjustment. A high `terminal_count` relative to `total_entries` signals excessive churn. See `references/data-shapes.md` for details.

---

## Pipeline States

All states are **lowercase**. Discovery operates on: `discovered → shortlisted` (approve) and `discovered → withdrawn` (reject). `reject_opportunity` transitions to `withdrawn` from any state that allows it. See `references/data-shapes.md` for the full state machine.
