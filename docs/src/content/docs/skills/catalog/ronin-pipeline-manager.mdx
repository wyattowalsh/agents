## Critical Rules

1. Always validate transitions against the state machine before calling `transition_pipeline` — show allowed transitions on error
2. Never execute bulk operations (approve, reject, withdraw) without explicit user confirmation
3. All state names must be lowercase — the DB constraint enforces this
4. Always show the `allowed` transitions list from the error response when a transition is invalid
5. Must distinguish `rejected` (by company) from `withdrawn` (by user) — `reject_opportunity` always transitions to `withdrawn`
6. Never transition entries in terminal states (`accepted`, `rejected`, `withdrawn`, `archived`) except `archived` which has no transitions
7. Always include a `note` with transitions to maintain an audit trail
8. Must present stale entry recommendations to the user before bulk withdrawal — stale does not mean abandoned
9. Funnel counts are cumulative — never present them as exclusive per-stage counts
10. `approve_opportunity` only works from `discovered` state — use `transition_pipeline` for entries past `discovered`

Manage the Ronin application pipeline end-to-end: review funnel health, triage discovered opportunities in bulk, transition entries between states, detect stale entries, and determine next actions across the full 12-state lifecycle.

> **Step 0 — Context Load (all skills):** Before any workflow, load `get_preferences(user_id)` and `get_user_memory(user_id)`. Load `get_pipeline(user_id)` when you need a full pipeline snapshot.

## Request Router

Classify the ask before choosing tools:

- **Overview / status / health** — Workflow 1 for read-only funnel and bottleneck analysis.
- **Triage / process discovered opportunities** — Workflow 2. Present recommendations first unless the user already gave explicit approve/reject instructions.
- **Next / next-actions / what should I work on next** — Workflow 3 for a read-only queue.
- **Stale / stuck / inactive** — Workflow 4 for read-only stale detection first.
- **Move / advance / transition** — Workflow 5 when the user names a specific entry and target state.

Default to read-only analysis when the request is ambiguous. Only call mutating tools (`approve_opportunity`, `reject_opportunity`, `transition_pipeline`) when the user explicitly asked for the mutation or confirmed your proposed action set.

## Overview

This skill provides a **cross-cutting view** of the entire pipeline — unlike other ronin-* skills that focus on a single stage (discovery, research, materials), this skill operates across all 12 states simultaneously. It answers questions like:

- "What does my pipeline look like right now?"
- "Where are my bottlenecks?"
- "Which entries are stuck?"
- "What should I work on next?"

### Scaling strategy

- `get_pipeline(user_id)` is the authoritative snapshot; start there before issuing per-entry follow-up calls.
- `list_opportunities(...)` is capped at 50 results. If the discovered queue exceeds that, say so explicitly and batch by `state`, `min_score`, or user-approved slices instead of implying full coverage.
- For stale detection on large active pipelines, batch `get_pipeline_history` calls and surface progress. Do not silently skip entries because history reads are expensive.
- For bulk triage, present batches by score tier and confirm each mutating batch before approve/reject calls unless the user already supplied exact instructions.

## State Machine Reference

All states are **lowercase** (matching the DB constraint). The pipeline has 12 states with directed transitions:

| From State | Allowed Transitions |
|---|---|
| `discovered` | `shortlisted`, `rejected`, `withdrawn` |
| `shortlisted` | `approved`, `rejected`, `withdrawn` |
| `approved` | `materials_generated`, `withdrawn` |
| `materials_generated` | `applied`, `withdrawn` |
| `applied` | `phone_screen`, `rejected`, `withdrawn` |
| `phone_screen` | `interview_loop`, `rejected`, `withdrawn` |
| `interview_loop` | `offer`, `rejected`, `withdrawn` |
| `offer` | `accepted`, `rejected`, `withdrawn` |
| `accepted` | `archived` |
| `rejected` | `archived` |
| `withdrawn` | `archived` |
| `archived` | (none) |

### State categories

**Active forward states** (entries progressing through the pipeline):
`discovered`, `shortlisted`, `approved`, `materials_generated`, `applied`, `phone_screen`, `interview_loop`, `offer`

**Terminal states** (entries that have reached an end state):
`accepted`, `rejected`, `withdrawn`, `archived`

### Funnel stage mapping

The `get_funnel_stats` tool maps the 12 pipeline states into 5 funnel buckets for conversion analysis:

| Funnel Bucket | Pipeline States |
|---|---|
| `discovered` | `discovered`, `shortlisted` |
| `approved` | `approved`, `materials_generated` |
| `applied` | `applied` |
| `interview` | `phone_screen`, `interview_loop` |
| `offer` | `offer`, `accepted` |

Terminal states (`rejected`, `withdrawn`, `archived`) are excluded from funnel counts and tracked separately via `terminal_count`.

Funnel counts are **cumulative** — each stage includes entries that have passed through it into deeper stages. For example, `discovered_count` includes entries currently in discovered + all entries that have progressed beyond discovered.

Load `references/tool-reference.md` for full tool signatures, parameters, and response shapes.

Summary of available tools:

- `get_pipeline(user_id, state?)` — Retrieve all pipeline entries, optionally filtered by state
- `get_pipeline_entry(user_id, entry_id)` — Fetch a single entry with linked opportunity and last transition

- `get_funnel_stats(user_id)` — Cumulative funnel conversion statistics

- `reject_opportunity(user_id, opportunity_id)` — Reject (transitions to `withdrawn`)
- `compute_match_score(user_id, opportunity_id)` — Re-compute match score (0-100, lexical cosine similarity)

## Workflow 1: Pipeline Overview

Get a holistic view of the pipeline with conversion rates and bottleneck analysis.

### Step 0 — Context load

```python
# Parallel context load
prefs = mcp__ronin__get_preferences(user_id="<uuid>")
memory = mcp__ronin__get_user_memory(user_id="<uuid>")
```

Extract personalization signals from `memory` and target context from `prefs` to inform recommendations.

### Step 1 — Fetch pipeline and funnel stats

Issue both calls in parallel:

```python
pipeline = mcp__ronin__get_pipeline(user_id="<uuid>")
funnel = mcp__ronin__get_funnel_stats(user_id="<uuid>")
```

### Step 2 — Group entries by state

From `pipeline.entries`, build a grouped view showing title, company, match_score per entry.

### Step 3 — Analyze conversion rates

Present the funnel metrics from `funnel.conversion_rates` (Discovered → Approved → Applied → Interview → Offer).

### Step 4 — Identify bottlenecks

A bottleneck is a stage with disproportionately low conversion. Apply these heuristics:

| Condition | Diagnosis | Recommendation |
|---|---|---|
| `approved_per_discovered < 0.3` | Low triage throughput | Run Bulk Triage (Workflow 2) to process discovered entries |
| `applied_per_approved < 0.4` | Materials generation stall | Run `ronin-materials` on approved entries |
| `interview_per_applied < 0.3` | Low response rate | Review application quality; consider reaching out via `ronin-networking` |
| `offer_per_interview < 0.2` | Interview conversion issue | Run `ronin-interview-prep` for active interview entries |
| Many entries in terminal states | High attrition | Review reject/withdraw patterns for preference refinement signals |

Present bottleneck findings with specific entry counts and actionable recommendations.

## Workflow 2: Bulk Triage

Process discovered opportunities in batches, approving or rejecting based on match score and user preferences.

### Step 1 — Load triage queue

```python
discovered = mcp__ronin__list_opportunities(
    user_id="<uuid>",
    state="discovered",
    min_score=30,
    sort_by="match_score"
)
```

If `discovered.count == 0`, report that the triage queue is empty and suggest running `ronin-discovery` to populate it.

### Step 2 — Present in score tiers

Group opportunities by match score tier:

| Score | Tier | Guidance |
|---|---|---|
| `>= 60` | Strong match | Present first; strong approve signal |
| `30-59` | Moderate match | Worth reviewing; apply judgment on seniority/location/industry fit |
| `< 30` | Weak match | Not loaded by default (filtered by `min_score=30`); mention count if user asks |

For each opportunity, show: title, company, location, remote status, match score and tier, URL.

Offer batch operations: "Approve all strong matches?" or "Review one-by-one?"

### Step 3 — Execute approve/reject decisions

**Approve** (transitions `discovered` to `shortlisted`): `mcp__ronin__approve_opportunity(user_id, opportunity_id)`

**Reject** (transitions to `withdrawn`): `mcp__ronin__reject_opportunity(user_id, opportunity_id)`

If the user asked for triage but did not explicitly authorize which entries to mutate, stop after presenting recommendations and get confirmation on the proposed approvals/rejections.

Process in batches. After each batch, report progress: "Approved 5, rejected 3, 6 remaining."

### Step 4 — Summarize triage session

After all decisions, summarize patterns:
- Count of approved vs. rejected
- Common traits of approved entries (e.g., "3 of 5 approved were fintech companies")
- Common traits of rejected entries (e.g., "4 of 6 rejected had match_score < 40")
- Suggest preference adjustments if patterns emerge (e.g., "You consistently reject devtools roles — should I remove 'devtools' from your industries?")

## Workflow 3: Next-Action Queue

For each active (non-terminal) pipeline entry, determine which ronin skill should be run next based on its current state.

### Step 1 — Load active entries

```python
pipeline = mcp__ronin__get_pipeline(user_id="<uuid>")
```

Filter to non-terminal entries (exclude `accepted`, `rejected`, `withdrawn`, `archived`).

### Step 2 — Map entries to next actions

| Current State | Next Action | Skill to Run |
|---|---|---|
| `discovered` | Triage (approve/reject) | `ronin-pipeline-manager` (Workflow 2) |
| `shortlisted` | Run deep research, then generate materials | `ronin-research` + `ronin-materials` |
| `approved` | Generate application materials for legacy/manual flows | `ronin-materials` |
| `materials_generated` | Submit application, then transition to applied | Manual + `transition_pipeline` |
| `applied` | Monitor for recruiter response; transition on signal | `ronin-email-monitor` |
| `phone_screen` | Prepare for phone screen | `ronin-interview-prep` |
| `interview_loop` | Prepare for interview rounds | `ronin-interview-prep` |
| `offer` | Evaluate and negotiate offer | `ronin-offer-negotiation` |

### Step 3 — Present prioritized queue

**Immediate action needed** — entries in states with clear next steps and no blockers:
- `discovered` entries awaiting triage
- `shortlisted` entries ready for research/materials generation
- `approved` entries ready for legacy/manual materials generation
- `offer` entries requiring evaluation

**Monitoring** — entries in states where external events drive progress:
- `applied` entries awaiting response
- `phone_screen` and `interview_loop` entries with upcoming interviews

**Research needed** — entries that need deeper investigation before progressing:
- `shortlisted` entries pending research

For each entry, show the opportunity title, company, current state, and the recommended next action.

## Workflow 4: Stale Entry Detection

Identify entries that have not had a state transition in 14 or more days.

### Step 1 — Load all active entries

```python
pipeline = mcp__ronin__get_pipeline(user_id="<uuid>")
```

Filter to non-terminal entries.

### Step 2 — Check transition history for each entry

For each non-terminal entry, fetch its history:

```python
history = mcp__ronin__get_pipeline_history(user_id="<uuid>", entry_id="<pe-uuid>")
```

The most recent transition is the **last item** in the `transitions` array (ordered by `created_at` ascending). Compare `created_at` of the last transition to the current date.

> **Performance note:** This step requires one `get_pipeline_history` call per active entry. For large pipelines (20+ active entries), this may result in many sequential calls. Issue calls in parallel when possible (model-native parallel tool calling).

### Step 3 — Flag stale entries

An entry is **stale** if its most recent transition `created_at` is 14+ days ago and it is in a non-terminal state.

| Days Since Last Transition | Severity | Recommended Action |
|---|---|---|
| 14-30 days | Stale | Review and either advance or withdraw |
| 30+ days | Very stale | Strongly consider withdrawing unless actively waiting |

For each stale entry, show: opportunity title and company, current state, days since last transition, last transition details, recommended action based on state (see Workflow 3 mapping).

### Step 4 — Offer bulk cleanup

If multiple entries are very stale (30+ days), offer to withdraw them in bulk:

```python
mcp__ronin__transition_pipeline(
    user_id="<uuid>",
    entry_id="<pe-uuid>",
    to_state="withdrawn",
    note="Withdrawn: no activity for 30+ days"
)
```

Always confirm with the user before bulk withdrawal — stale does not always mean abandoned.

## Workflow 5: Pipeline Transition

Transition a single pipeline entry to a new state with validation.

### Step 1 — Verify current state

```python
entry = mcp__ronin__get_pipeline_entry(user_id="<uuid>", entry_id="<pe-uuid>")
```

Confirm the entry's `state` field and check `last_transition` for context. Display the current state and allowed transitions (from the State Machine Reference) to the user.

### Step 2 — Validate the target state

Before calling `transition_pipeline`, verify the requested `to_state` is in the allowed transitions for the current state. If not, inform the user with the allowed transitions list.

### Step 3 — Execute the transition

```python
mcp__ronin__transition_pipeline(
    user_id="<uuid>",
    entry_id="<pe-uuid>",
    to_state="phone_screen",
    note="Recruiter call scheduled for Friday 3pm"
)
```

The `note` parameter is optional but recommended. It creates an audit trail in the transition history. Encourage users to provide context for transitions (e.g., "Interview scheduled for March 25", "Rejected: ghosted after 3 weeks", "Withdrew: comp below expectations").

If the user is still deciding, keep the request advisory and do not call `transition_pipeline` yet. Use the allowed-transitions list to explain the options first.

### Step 4 — Confirm and suggest next steps

After a successful transition, confirm the change and recommend the appropriate next action based on the new state. Use the state-to-skill mapping from Workflow 3 to determine the recommendation.

Use transition notes to preserve audit context, especially when moving entries for reasons that will matter later (rejection reason, interview timing, withdrawal rationale).
