# ronin-research

| `$ARGUMENTS` | Action |
|------------|--------|
| `research <company> <role>` or opportunity reference | Run Research workflow |
| `review <company>` or `status <opportunity>` | Review Research workflow |
| `refresh <company>` | Force re-research (bypass cache) |
| Empty | Show research status for all shortlisted opportunities |

A skill for deep-diving on a company and role once an opportunity is worth pursuing in Ronin. It uses a 3-layer model: shared company research, shared role research, and per-opportunity research persisted through `create_research`. Company and role layers should be reused when possible; user-fit analysis should be refreshed from the current opportunity and profile context.

> **Step 0 — Context Load:** Start by loading `get_preferences(user_id)`, `get_user_memory(user_id)`, and, when you already have an opportunity, `get_opportunity(user_id, opportunity_id)` in parallel.

---

## Overview

Research is best run once an opportunity has reached `shortlisted` or later. The MCP research tools verify ownership through the pipeline entry; they do not themselves enforce a specific minimum state.

### Cache-First Principle

**Always check existing data before making web requests.** Company research and role research are global layers shared across users. Before running any search queries:

1. Call `get_company_research(company_slug=...)` — if `found: true`, reuse company-layer data.
2. Call `get_role_research(role_slug=...)` — if `found: true`, reuse role-layer data.
3. Optionally check `get_cached_company_data(company_name=...)` as a fast fallback for recent company payloads.
4. Only search for dimensions that are missing or need refreshing.

This avoids redundant API calls and respects rate limits on Brave/Tavily. Load `references/tool-reference.md` for cache key formats, TTL details, and force-refresh instructions.

### Slug Canonicalization

Company and role research layers are keyed by slugs, not display names.

**Company slugs** are kebab-case identifiers (e.g. `"stripe"`, `"jane-street"`).

**Role slugs** are produced by `canonicalize_role_slug()` on the server, which normalizes titles before storage and lookup:

1. Lowercase
2. Strip noise words: `software`, `platform`, `backend`, `frontend`, `full stack`, `fullstack`
3. Map synonyms: `swe`/`dev`/`developer` -> `engineer`
4. Standardize seniority: `sr` -> `senior`, `jr` -> `junior`
5. Convert to kebab-case

Examples: `"Staff Software Engineer"` -> `"staff-engineer"`, `"Sr. Backend Developer"` -> `"senior-engineer"`, `"Junior SWE"` -> `"junior-engineer"`, `"Principal Platform Engineer"` -> `"principal-engineer"`.

The `get_role_research` and `upsert_role_research` tools both apply this canonicalization automatically, so you may pass either the raw role title or the already-canonicalized slug.

### 3-Layer Data Model

Research data is stored across three persisted layers plus one fast cache layer:

1. **Company research** (global, PostgreSQL, keyed by `company_slug`) -- Check first via `get_company_research`. Contains `company_overview`, `tech_stack`, `culture_signals`, `compensation_benchmark`, `trajectory`, `interview_process`, `sources`. Upsert via `upsert_company_research` only when data is missing or stale.
2. **Role research** (global, PostgreSQL, keyed by `role_slug`) -- Check first via `get_role_research`. Contains `salary_p25`/`p50`/`p75`, `sample_size`, `common_requirements`, `typical_interview_stages`, `market_demand`, `sources`, `completeness_score`. Upsert via `upsert_role_research` only when data is missing or stale.
3. **Per-opportunity research** (per-opportunity, PostgreSQL) -- Stored via `create_research`. Contains the seven persisted research dimensions for a specific opportunity, including user-specific `requirements_match`. Always refresh the user-fit portion from current profile/opportunity context.
4. **Redis company cache** (fast cache, keyed by company name) -- Read via `get_cached_company_data`. `create_research` refreshes this cache after persisting the opportunity-layer record so future runs can avoid redundant company lookups.

The persistence pattern: check company and role research first, optionally consult the Redis cache for a recent company payload, search only for missing dimensions, upsert any refreshed company/role layers, then persist the per-opportunity record with `create_research`, which also refreshes the Redis company cache. Load `references/data-shapes.md` for full field-level schemas.

### Scaling strategy

Keep research runs proportional to what is actually missing:

- Group work by company and role when reviewing multiple shortlisted opportunities.
- Reuse fresh company and role layers across opportunities instead of repeating the same external searches.
- Treat the Redis company cache as a fast fallback after the persisted company and role layers, not as the primary authority.
- Only run external queries for the missing or explicitly refreshed dimensions.
- Use DeepWiki only when you have a real repo or org target that can improve stack or engineering-process depth.

### Research Dimensions

The research pipeline produces seven fields of intelligence for the per-opportunity record (`create_research`):

**Company overview** -- a narrative description of the company (2-3 sentences) synthesized from web search results.

**Tech stack** -- languages, frameworks, and infrastructure tooling categorized into `primary`, `secondary`, and `tools` arrays. Extracted from engineering blogs, job postings, public repos, and DeepWiki.

**Compensation benchmark** -- min/max/median salary estimates with currency and free-text notes about data sources. Compared against the user's configured compensation goals.

**Requirements match** -- a structured comparison of the role's required skills against the user's profile, producing `matched_skills`, `missing_skills`, and a `match_percentage`. Always computed fresh (user-specific).

**Culture signals** -- structured indicators including `work_style`, `team_size`, `growth_stage`, and free-text `notes`. Extracted from reviews and company pages.

**Red flags** -- a list of potential concerns identified during research (e.g. high turnover, negative Glassdoor trends, unclear role scope).

**Interview process** -- a free-text description of the typical interview format when available from Glassdoor reviews or community reports. `null` when data is unavailable.

### Synthesis

Use the model to synthesize search results into the seven persisted research dimensions. If helpful, you may track confidence in the summary you present to the user, but the current Ronin MCP contract does **not** persist a dedicated confidence field per dimension in `create_research`.

All persisted results are written to Ronin via `mcp__ronin__create_research` and linked to the opportunity record. Company and role layers are persisted via their respective upsert tools for cross-user reuse.

## Prerequisites

### 1. Opportunity ready for deeper review

Research is most useful for opportunities the user has already triaged into `shortlisted` or beyond. Run `ronin-discovery` first to populate the pipeline; the human-in-the-loop approval step is shortlisting a discovered opportunity. Ronin queues research from that shortlist action, then successful research queues materials generation and moves the entry to `materials_generated` when the draft is ready. Confirm current state with:

```python
mcp__ronin__get_opportunity(user_id="<user-id>", opportunity_id="<opp-uuid>")
# Typical workflow target: response.opportunity.state in {"shortlisted", "approved"}
```

If the opportunity is still in `discovered` state, treat that as a workflow warning rather than a tool-level blocker. The normal path is to triage it first.

### 2. Valid `opportunity_id`

Required for all research tool calls. Obtain it from the `opportunity.id` field of any `get_opportunity` or `list_opportunities` response.

### 3. External search tools configured

Brave, Tavily, and DeepWiki are optional enrichment sources used only when cached company and role layers are missing or stale. If Brave is unavailable, the skill can still proceed from Ronin data plus other search tools. Always check `get_company_research`, `get_role_research`, and optionally `get_cached_company_data` before using web search.

### 4. Ronin MCP tools available in-session

This skill assumes the current agent session already has access to the Ronin MCP tools listed in `allowed-tools`.

### 5. Valid `user_id`

Ronin MCP signatures keep `user_id: str` at the boundary. Pass the active Ronin user identifier exactly as supplied by the current environment. In the current local implementation, research MCP paths still validate UUID-shaped values, so treat format mismatches as runtime constraints rather than rewriting IDs manually.

## Workflow: Run Research on an Opportunity

Run this after the opportunity has been shortlisted.

### Step 1 -- Self-fetch context

The agent loads its own context at startup. These calls can be issued in parallel:

```python
# Parallel self-fetch
opp = mcp__ronin__get_opportunity(user_id="<user-id>", opportunity_id="<opp-uuid>")
prefs = mcp__ronin__get_preferences(user_id="<user-id>")
memory = mcp__ronin__get_user_memory(user_id="<user-id>")
```

Extract the fields needed to drive research:

- `opp.opportunity.company` -- used as the cache key and search subject
- `opp.opportunity.title` -- used to build role-specific search queries
- `opp.opportunity.url` -- primary source for role analysis; fetch the full description if the stored snippet is short
- `opp.opportunity.description` -- the job description text already stored in Ronin; use this for role analysis before falling back to web searches
- `opp.opportunity.required_skills` -- used for requirements matching
- `prefs.preferences.compensation_goals` -- used for compensation comparison
- `memory` -- personalization signals for user fit analysis

### Step 2 -- Check existing research layers (CACHE-FIRST)

**This step is critical.** Before making ANY web requests, check whether company and role data already exist. Skipping this step wastes API quota and produces duplicate data.

Derive slugs from the opportunity loaded in Step 1:

- **company_slug**: kebab-case the company name (e.g. `"Stripe"` -> `"stripe"`, `"Jane Street"` -> `"jane-street"`)
- **role_slug**: apply `canonicalize_role_slug()` logic to the title (e.g. `"Staff Software Engineer"` -> `"staff-engineer"`) -- see Slug Canonicalization above

```python
# Parallel layer checks -- use kebab-case slugs, not display names
company_data = mcp__ronin__get_company_research(company_slug="stripe")
role_data = mcp__ronin__get_role_research(role_slug="staff-engineer")
```

Determine which dimensions need fetching:

- If company_data exists and is fresh: skip company-level queries (1, 3, 5)
- If role_data exists and is fresh: skip role-level queries (2, 4, 6)
- If both exist: skip directly to user fit analysis (Step 4)
- If neither exists: run all 6 queries
- If the user explicitly asked for `refresh`, bypass the reusable layers intentionally and rebuild them from fresh search evidence

### Step 3 -- Search for missing dimensions (6 parallel queries)

Issue only the needed queries in parallel (model-native parallel tool calling). Do not imply that all six queries run on every pass; run only the dimensions not already covered by fresh reusable data:

**Query 1 -- Company background:**

```python
mcp__tavily__tavily_search(
    query="{company} company background products customers",
    max_results=5
)
```

**Query 2 -- Tech stack depth (DeepWiki when you have a real repo/org target):**

```python
mcp__deepwiki__ask_question(
    repoName="org-or-repo",
    question="What languages, frameworks, or infrastructure technologies does {company} publicly discuss for {role}-adjacent engineering work?"
)
```

**Query 3 -- Company culture:**

```python
mcp__tavily__tavily_search(
    query="{company} company culture remote work environment",
    max_results=5
)
```

**Query 4 -- Compensation:**

```python
mcp__brave-search__brave_web_search(
    query="{company} engineer salary compensation levels",
    count=5
)
```

**Query 5 -- Recent news and funding:**

```python
mcp__tavily__tavily_research(
    input="Research recent news, funding, growth, and notable changes for {company} relevant to evaluating this opportunity."
)
```

**Query 6 -- Interview process:**

```python
mcp__brave-search__brave_web_search(
    query="{company} {role} interview process hiring",
    count=5
)
```

### Step 4 -- Synthesize research data

Extended thinking handles the synthesis. The model receives all search results, existing company/role data, and the user's profile context.

The model produces the seven research dimensions (`company_overview`, `tech_stack`, `compensation_benchmark`, `requirements_match`, `culture_signals`, `red_flags`, `interview_process`). You may include confidence in your user-facing summary, but it is not a dedicated persisted field in the current `create_research` contract. Do NOT add CoT scaffolding.

Use the job description from the opportunity record as the primary source for `requirements_match` (user fit analysis -- always computed fresh). Apply these synthesis rules:

- **Null over confabulation:** Set numeric fields (compensation min/max/median, match_percentage) to `null` when no source data supports a specific value. A known gap is safer than a plausible-sounding hallucination.
- **Source attribution in notes:** The `compensation_benchmark.notes` and `culture_signals.notes` fields must contain the data source (e.g. "levels.fyi", "Glassdoor", "job posting"), approximate sample size or confidence qualifier (e.g. "n=47", "limited data"), and recency (e.g. "2024 data").
- **Populate what you can:** Fill every field from search results when supported. Use `null` for fields where data is genuinely unavailable rather than guessing.
- **Confidence in presentation:** If you surface confidence to the user, treat it as an agent-side summary aid based on source quality and recency rather than a persisted MCP field.

### Step 5 -- Store all 3 layers

Store research across all three layers. Company and role layers are upserted only if they were freshly fetched (not already present from Step 2). User fit analysis is always stored. Load `references/tool-reference.md` for full tool call examples with response shapes.

**Layer 1 -- Company research (if freshly fetched):** Call `upsert_company_research(company_slug, data)` with company_name, company_overview, tech_stack, culture_signals, compensation_benchmark, trajectory, interview_process, and sources.

**Layer 2 -- Role research (if freshly fetched):** Call `upsert_role_research(role_slug, data)` with title_variants, salary percentiles, common_requirements, typical_interview_stages, market_demand, sources, and completeness_score.

**Layer 3 -- Full research record (always):** Call `create_research(user_id, opportunity_id, data)` with all seven research dimensions. This upserts on `opportunity_id` and also refreshes the Redis company cache for the company.

State management: resume from persisted company research, persisted role research, or the Redis company cache when available. `create_research` is not just a cache write — it is the per-opportunity persistence step plus the cache refresh step.

### Step 6 -- Return structured JSON summary

Produce a structured summary with `company`, `role`, `layers_fetched`, `layers_cached`, and a `dimensions` map showing status and confidence for each of the seven fields. `layers_fetched` and `layers_cached` are reporting labels in your summary only; they are not persisted MCP fields.

Then present the key findings in a human-readable summary (see **Workflow: Review Research** for the recommended presentation format). Do not dump the raw JSON at the user -- synthesize it.

## Workflow: Review Research

Run this to present findings to the user after research has been stored, or to surface existing research for an opportunity already researched in a prior session.

### Step 1 -- Retrieve stored research

```python
mcp__ronin__get_research(
    user_id="<user-id>",
    opportunity_id="<opp-uuid>"
)
```

Returns `status: "researched"` with the full research record, or `status: "not_researched"` with `research: null`. Load `references/tool-reference.md` for full response shapes.

### Step 2 -- Load compensation goals

Retrieve the user's preferences to compare benchmarks against their targets:

```python
mcp__ronin__get_preferences(user_id="<user-id>")
```

### Step 3 -- Present findings

Structure the summary in five sections:

**Company Overview**
The `company_overview` narrative. Tech stack categorized by primary/secondary/tools. Culture signals: work style, team size, growth stage, and notes.

**Requirements Match**
Matched skills vs. missing skills. Match percentage. Flag any critical gaps that should be addressed in application materials.

**Compensation Analysis**
Min / median / max salary range from `compensation_benchmark`. Compare directly against `compensation_goals[0].min_base` from preferences -- flag clearly if median is below the user's minimum. Include notes about data sources and confidence.

**Interview Process**
The `interview_process` string. Flag any atypical patterns (e.g. 8+ rounds, unusual formats). If `null`, note that interview data is unavailable.

**Red Flags**
List any `red_flags` entries. If empty, note that no red flags were identified.

### Step 4 -- Make a recommendation

End the review with a clear recommendation in one of two forms:

**Advance:** "Strong match -- recommend generating materials. Median comp ($250k base) aligns with your $200k floor. Tech stack overlaps on Python and distributed systems. Standard 5-round process."

**Flag concerns:** "Concerns before advancing: (1) Median base ($180k) is below your $200k minimum -- consider clarifying comp expectations before investing application time. (2) Missing Rust skill -- flag this explicitly in application materials. (3) Red flag: high engineering turnover reported."

Do not automatically change the opportunity state -- present the recommendation and let the user decide.
