# ronin-interview-prep

| `$ARGUMENTS` | Action |

A skill for generating structured interview preparation once an opportunity reaches `phone_screen` or `interview_loop` state in Ronin. Produces STAR-format behavioral answers grounded in SSOT achievements, a prioritized technical study plan derived from research and role data, and company-specific talking points with questions to ask. Extended thinking (opus + 1M budget) handles the synthesis across profile, research, and role data. Every behavioral answer traces to a specific `achievement.id` — no fabricated stories.

> **Context load:** Start from the Ronin MCP connection already configured in the current harness. Load profile, preferences, memory, and any opportunity-specific context in parallel. Do not require a local `localhost:8000` health check unless the user is explicitly debugging local infrastructure.

## Three Pillars of Prep

1. **Behavioral prep** — likely behavioral questions mapped to SSOT achievements using STAR format. Each answer traces to a specific `achievement.id` from the profile. If the SSOT lacks a relevant achievement for a predicted question, the gap is acknowledged rather than filled with fabricated content.

2. **Technical prep** — a prioritized study plan that cross-references the role's `required_skills` and company `tech_stack` against the user's profile. Skills the user already has are marked as review items. Skills from `requirements_match.missing_skills` are marked as priority study areas. Web search supplements with company-specific interview patterns.

3. **Talking points and questions to ask** — synthesized from `company_overview`, `culture_signals`, and `trajectory` data in the company research. Produces 5 thoughtful questions the user can ask interviewers, grounded in real company context.

Extended thinking handles the synthesis step. Do NOT add CoT scaffolding — the reasoning model handles this internally.

---

## Prerequisites

1. **Opportunity in `phone_screen` or `interview_loop` state** — confirm via `get_opportunity`. If earlier, the user must advance the pipeline first.
2. **Completed research** — `get_research` must return `status: "researched"` with non-null `research`. If missing, run `ronin-research` first.
3. **Populated SSOT profile** — `get_profile` must return non-empty `experiences` with at least one achievement. If incomplete, run `ronin-profile-builder` first. A sparse profile produces sparse prep — flag this.
4. **`BRAVE_API_KEY`** — required for web searches. 401/403 means missing/invalid key. Obtain at [brave.com/search/api](https://brave.com/search/api).
5. **Ronin MCP connection available** — this skill assumes the current AI harness can already call Ronin MCP tools. If those calls fail, surface the MCP connection/configuration issue for the current environment.

## Workflow: Generate Interview Prep

### Step 1 — Self-fetch context (parallel)

All 7 calls are independent — issue in parallel. Load `references/tool-reference.md` for full tool signatures and key fields used from each response.

#### Validation gates

Before proceeding, verify:

1. `opp.opportunity.state` is `"phone_screen"` or `"interview_loop"` — if not, stop and inform the user
2. `research.status` is `"researched"` and `research.research` is non-null — if not, run `ronin-research` first
3. `profile.experiences` is non-empty and contains at least one achievement — if not, run `ronin-profile-builder` first

### Step 2 — Extract interview intelligence

Gather structured interview signals from the research layers:

**From `research.interview_process`** (free-text string):
- Parse the number of rounds and their types
- Identify which rounds are behavioral, technical, system design, and recruiter/logistics

**From `role.typical_interview_stages`** (list of strings):
- Cross-reference against `research.interview_process` to fill gaps
- If `interview_process` is `null`, use `typical_interview_stages` as the primary signal

**From `company.culture_signals`** and **`company.trajectory`**:
- Extract values and team dynamics relevant to behavioral questions
- Extract company direction, recent developments, strategic priorities for talking points

**Web search for company-specific patterns** (parallel): Search Brave for `{company} {role} interview questions` and Tavily for `{company} engineering interview system design topics`.

If web searches return useful patterns, incorporate them. If not, proceed with research layers alone.

### Step 3 — Generate behavioral prep

#### Question prediction

Predict 8-12 likely behavioral questions grouped by theme:

- **Leadership and influence** — "Tell me about a time you led a cross-functional initiative"
- **Technical decision-making** — "Describe a time you made a difficult technical trade-off"
- **Conflict and collaboration** — "Tell me about a time you disagreed with a teammate"
- **Growth and scaling** — "Describe a time you scaled a system or team"
- **Failure and learning** — "Tell me about a time something went wrong"

Customize based on role level, company culture signals, and interview format.

#### Achievement mapping

For each predicted question:

1. Scan all `experiences[].achievements[]` for relevance to the question theme
2. Rank by `impact_score` when multiple achievements are relevant
3. Select the single best match and record its `achievement.id`
4. Use `achievement.raw_text` as the factual anchor for the answer

#### STAR-format answer generation

For each question-achievement pair, generate a STAR answer sourcing fields as follows:

| STAR field | Source from achievement record | If source field is null |
|---|---|---|
| Situation | `context`, `baseline` | Use available context from `raw_text`; keep brief |
| Task | `action` | Derive from `raw_text` |
| Action | `action`, `raw_text` | Use `raw_text` as-is |
| Result | `result`, `metric` | State the outcome from `raw_text` without adding specifics |

**Critical constraints:**

- STAR sections must draw ONLY from fields present in the achievement record
- If a STAR field cannot be populated (e.g. `baseline` is null), use what is available and note the gap — do NOT invent context
- If the SSOT has no achievement relevant to a predicted question, mark it as **"GAP — no matching SSOT achievement"** and suggest the user prepare from memory or add the achievement via `ronin-profile-builder`
- Each answer must cite the `achievement.id` it traces to

#### Reuse policy

A single achievement may map to multiple questions if genuinely the best match. Flag when an achievement is reused across 3+ questions — this indicates SSOT sparsity, and the user should consider enriching their profile.

### Step 4 — Generate technical prep

#### Skill gap analysis

Cross-reference these data sources:

- `opp.opportunity.required_skills` — what the role requires
- `research.research.tech_stack` — company's technology landscape
- `research.research.requirements_match.matched_skills` / `.missing_skills`
- `role.common_requirements` — typical requirements for this role level
- `profile.skills` — the user's full skill inventory

Categorize each skill:

| Category | Definition | Prep action |
|---|---|---|
| **Strong match** | In both `required_skills` and user's `skills` | Light review; prepare to discuss depth |
| **Partial match** | Related skill in profile but not exact | Moderate review; brush up on specifics |
| **Gap** | In `required_skills` / `missing_skills` but not in profile | Priority study area |
| **Nice-to-have** | In `tech_stack` but not `required_skills` | Low priority; mention familiarity if genuine |

#### Study plan

For each gap and partial-match skill, produce: Priority (HIGH/MEDIUM/LOW), skill name, why it matters, study plan with time estimate, and resources from web search if available.

#### System design topics

If the interview includes a system design round, search for company-specific topics and produce 3-5 relevant topics. For each: the problem statement, relevance to the company's domain, key components, relevant tech from the company's stack, and how the user's experience connects.

#### Coding round prep

If the interview includes coding rounds, search for company-specific patterns and provide guidance on difficulty level, question types, preferred languages, and time management.

### Step 5 — Generate talking points and questions to ask

**Talking points** (5-8): Synthesize from `company_overview`, `culture_signals`, `trajectory`, and `compensation_benchmark` (internal reference only). Each 1-2 sentences, grounded in specific research data.

**Questions to ask** (exactly 5): Each must be grounded in specific research findings (cite the data source), demonstrate genuine company knowledge, be relevant to the role level, and not be easily answerable from the job posting.

### Step 6 — Present structured output

Present in three clearly separated sections. Do not dump raw JSON — synthesize into readable format.

**Section 1: Behavioral Prep** — header with company/role/interview format, then each question with: achievement ID (or GAP), STAR answer, source citation. End with achievement reuse summary and gap count.

**Section 2: Technical Prep** — skill assessment table, priority study plan with time estimates, system design topics (if applicable), coding round guidance (if applicable).

**Section 3: Talking Points and Questions** — numbered talking points with sources, then 5 questions to ask with source citations.

Load `references/data-shapes.md` for detailed output field schemas.

## Critical Constraint: No Fabrication

> **Red line**: NEVER generate behavioral answers that include achievements, metrics, or claims not present in the SSOT. Every STAR answer must trace to a specific `achievement.id` whose `raw_text` contains the facts being presented.

| SSOT contains | You MAY write | You MUST NOT write |
|---|---|---|
| `raw_text`: "Led migration of monolith to microservices, enabling 5x deployment frequency" | "I led a monolith-to-microservices migration that enabled 5x deployment frequency" | "I led a migration that reduced costs by 40% and improved reliability to 99.99%" |
| `metric`: null, `raw_text`: "Improved API performance through caching" | "I improved API performance through a caching strategy" | "I improved API performance by 60% through Redis caching" |
| No achievement about conflict resolution | Mark as GAP; suggest user prepare from memory | Invent a conflict resolution story |

## Workflow: Mock Interview

Interactive mock interview using generated prep as the question bank. Requires prep to exist (either generated this session or regenerated via Generate workflow).

1. **Setup** — inform the user of the format: "I'll ask behavioral and technical questions from your prep. After each answer, I'll evaluate against your SSOT-grounded prep."
2. **Question order** — (a) questions with strong SSOT achievements first (build confidence), (b) GAP questions (identify improvisation areas), (c) technical questions from the study plan. Wait for user response before proceeding.
3. **Evaluate each response** — check SSOT alignment (did they cite a real achievement? add details not in SSOT?), STAR structure quality, and provide specific feedback. Flag any details mentioned that are not in the SSOT — suggest adding via `ronin-profile-builder` if accurate.
4. **Summary** — after all questions: questions asked count, strong answers, needs-work items, GAPs identified, SSOT enrichment suggestions.

## Workflow: Review Existing Prep

Interview prep is generated on-demand (not stored persistently). Reviewing prep means regenerating from current state.

1. Follow Steps 1-2 from Generate Interview Prep to load context and extract intelligence.
2. Run Steps 3-6 to regenerate the full prep.
3. If the user has seen prep before, highlight changes driven by: new achievements added, updated research data, skills added/removed.
