## Critical Rules

1. Never apply pipeline transitions without user confirmation — `confirm_signal` is always required.
2. Only store signals with confidence >= 0.6 — below threshold signals are discarded.
3. Skip emails that are not job-search-related entirely — do not create signals for irrelevant emails.
4. Gmail OAuth uses read-only scope — never send, modify, or delete emails.
5. Never log or include OAuth tokens in error messages or API responses.
6. Use `search_gmail` or `read_gmail` as bounded connection checks before processing Gmail content.
7. Use the classification decision tree for manual sync — do not invent signal types.
8. `confirm_signal` may mark a signal acted on without moving the pipeline when no valid entry is available or the current entry is already at/beyond the signal-implied state.

## Overview

This skill monitors Gmail for emails that carry career-relevant signals. For every email it processes, it determines a `signal_type` and `confidence` score, then suggests the appropriate pipeline state transition. The user confirms or dismisses each suggestion — no pipeline state changes happen automatically.

### Signal types

| Signal Type | Description |
|-------------|-------------|
| `confirmation` | ATS or employer confirms receipt of application |
| `rejection` | Employer declines to move forward |
| `interview_invite` | Employer invites candidate to a screen or interview |
| `offer` | Formal or informal offer extended |
| `other` | Unsolicited recruiter contact, informational update, or other career-related email |

Emails that are not job-search-related should be skipped entirely — do not store a signal.

### What this skill actually does

This skill is a manual/operator workflow over the live Gmail and Ronin MCP tools. It helps you search Gmail, read candidate emails, classify them, and store reviewable signals. Product surfaces may invoke the same backend, but this skill should only claim behavior visible through the exposed MCP tools.

`create_email_signal` stores Gmail metadata, a `signal_type`, a confidence score, and an optional `suggested_transition`. Linking and normalization happen at confirmation time. `confirm_signal` accepts an optional `pipeline_entry_id` override and an optional normalized `opportunity_data` payload for unmatched signals, so reviewed signals can either attach to an existing entry or create a researchable opportunity directly in the intended state.

### Request router and confirmation contract

Route requests by operator intent:

- **`connect`** — Run the Connect Gmail workflow.
- **`sync` / `check` / scan-my-inbox style requests** — Run Manual Sync.
- **`signals` / `pending` / what-needs-review requests** — List pending signals first.
- **Empty** — Check token status and pending-signal count only.

Keep confirmation policy consistent across all routes:

- **Below 0.6 confidence** — skip; do not store a signal.
- **0.6-0.75 confidence** — show the evidence and get user agreement before creating the signal.
- **Above 0.75 confidence** — create the signal, then present it for confirm/dismiss review.
- **Above 0.95 confidence** — prioritize it in the review queue as an obvious signal, but still do not auto-confirm it.

`confirm_signal` is the only mutation gate for reviewed signals. It may:

- apply the suggested pipeline transition when the stored or supplied pipeline entry is valid
- create a normalized opportunity plus pipeline entry when the signal is unmatched and `opportunity_data` is supplied
- acknowledge the signal without moving the pipeline when the current state is already equal to or later than the signal-implied state

## Prerequisites

### 1. Gmail OAuth connected

Run the Connect Gmail workflow below once per account. Until tokens are stored, no Gmail access is possible.

Use `search_gmail` or `read_gmail` as the public connection check. If either returns `gmail_not_connected`, run the Connect Gmail workflow.

### 2. Active pipeline entries

The email monitor does not auto-link signals from sender domains or company-name guesses. Live confirmation can only act through an already linked `signal.pipeline_entry_id`, an explicit `pipeline_entry_id` override, or reviewed normalized `opportunity_data`. An empty pipeline therefore produces unmatched signals that the user must manually assign or materialize.

Populate the pipeline first using `ronin-discovery` or direct pipeline management actions (for example `ronin-pipeline-manager` triage/transition workflows).

### 3. Gmail OAuth environment variables

```bash
export GMAIL_CLIENT_ID="<google-oauth-client-id>"
export GMAIL_CLIENT_SECRET="<google-oauth-client-secret>"
export FERNET_KEY="<base64-fernet-key>"  # generate: python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
```

Obtain Gmail credentials from Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client IDs. The redirect URI must match what is configured in the `gmail_redirect_uri` setting (default: `http://localhost:8000/auth/gmail/callback`).

### 4. Ronin MCP tools available in-session

This skill assumes the current agent session already has access to the Ronin MCP tools listed in `allowed-tools`.

### 5. Valid `user_id`

Ronin MCP signatures keep `user_id: str` at the boundary. Pass the active Ronin user identifier exactly as supplied by the current environment. In the current local implementation, some Gmail and signal paths still validate UUID-shaped values, so treat format mismatches as runtime constraints rather than rewriting IDs manually.

## Workflow: Connect Gmail

Run once per account. Tokens are stored encrypted and automatically refreshed.

### Step 1 — Initiate OAuth

Call `mcp__ronin__connect_gmail(user_id="<user-id>")`. Returns an `oauth_url` for the user to open in a browser, or `error: "gmail_not_configured"` if env vars are missing.

### Step 2 — User authorizes

Present the `oauth_url` to the user. They must open it in a browser and authorize Ronin. The OAuth scope is `gmail.readonly` — Ronin requests read-only access and never sends emails.

After authorization, Google redirects to the configured callback URI with an authorization code. The callback handler exchanges it for tokens.

### Step 3 — Store tokens

In the normal product flow, the OAuth callback stores the tokens for you. Call `mcp__ronin__store_gmail_tokens(user_id, access_token, refresh_token, expires_at)` only when you are handling the OAuth exchange manually in an integration or test workflow. Tokens are encrypted at rest and never logged.

Load `references/tool-reference.md` for full parameter signatures and response shapes.

## Workflow: Manual Sync

Use this to immediately review recent Gmail messages and create signals from them.

### Step 1 — Search Gmail for career emails

Use either Ronin MCP tools (preferred — handles token refresh) or Gmail MCP tools:

```python
# Ronin MCP (works in all environments, auto-refreshes tokens)
mcp__ronin__search_gmail(
    user_id="<user-id>",
    query="is:unread newer_than:7d -category:promotions -category:social from:(hr@ OR recruiting OR talent OR careers) subject:(application OR interview OR offer OR unfortunately OR next steps)",
    max_results=50
)

# Gmail MCP (available when mcp__gmail__ server is connected)
mcp__gmail__search_emails(
    query="is:unread newer_than:7d -category:promotions -category:social from:(hr@ OR recruiting OR talent OR careers) subject:(application OR interview OR offer OR unfortunately OR next steps)",
    maxResults=50
)
```

**Important**: Always include `-category:promotions -category:social` to exclude marketing and social emails at the Gmail level. Do NOT include `-category:updates` — ATS platforms (Greenhouse, Lever, Workday) deliver legitimate emails through Gmail's Updates category.

For high-volume inboxes, narrow the Gmail query itself (`newer_than:3d`) or tighten the subject filter. Remove `noreply` from sender filters — it matches too many marketing emails. When the query still returns a large result set, process messages in batches of 10-20 and tell the user which slice you are reviewing instead of skimming or silently truncating.

### Step 2 — Read each email

```python
# Ronin MCP (preferred)
mcp__ronin__read_gmail(user_id="<user-id>", message_id="<gmail-message-id>")

# Gmail MCP (when available)
mcp__gmail__read_email(messageId="<gmail-message-id>")
```

Fields used for classification:

| Field | Used for |
|-------|---------|
| `subject` | Keyword detection |
| `from` | Sender identification |
| `body` | Full content classification |
| `snippet` | Quick preview for signal storage |

### Step 3 — Classify the email

Use the Classification section below to determine `signal_type` and `confidence`. Apply the confidence threshold:

- If `confidence >= 0.6`: proceed to Step 4 (create the signal).
- If `confidence < 0.6`: skip — do not store.

> **Decision Point**: For ambiguous emails (confidence 0.6–0.75), present the email to the user and ask them to verify the classification before creating the signal.
>
> **High-confidence review path**: For signals with confidence > 0.95 — especially rejections with unambiguous language — create the signal and present it at the top of the review queue with a strong recommendation to confirm. Do not auto-confirm it; `confirm_signal` is still required.

### Step 4 — Create the email signal

Call `mcp__ronin__create_email_signal(user_id, gmail_message_id, signal_type, subject, sender, confidence, suggested_transition, raw_snippet)`. Returns `status: "created"` or `status: "existing"` if already processed. This records the signal and suggested transition only; it does not directly link a pipeline entry.

Load `references/tool-reference.md` for full parameter signatures and response shapes.

### Step 5 — List pending signals

Call `mcp__ronin__list_email_signals(user_id, acted_on=False)` to retrieve unacted signals.

### Step 6 — User confirms or dismisses each signal

Present each signal using this format:

```text
[N] SIGNAL_TYPE — Company/Sender (confidence%)
    Subject: "..."
    Suggested: move to <state>
    → Confirm or dismiss?
```

Group signals by company when presenting multiple.

**Confirm** — `mcp__ronin__confirm_signal(user_id, signal_id, pipeline_entry_id=None, opportunity_data=None)` — uses one of three paths:

- linked or overridden entry: apply the suggested transition when valid
- unmatched + normalized payload: create `Opportunity` + `PipelineEntry` directly in `opportunity_data.initial_state`
- no valid link / no payload: mark the signal acted on without changing pipeline state

**Dismiss** — `mcp__ronin__dismiss_signal(user_id, signal_id)` — marks the signal as acted on without changing the pipeline.

When `confirm_signal` returns `transition_applied: false`, explain whether it was an acknowledge-only confirmation (already at or beyond the implied state), a missing-link case, or an invalid transition.

Load `references/tool-reference.md` for full response shapes and error codes.

## Signal Types and Pipeline Transitions

| Signal Type | Suggested Transition | Notes |
|-------------|---------------------|-------|
| `confirmation` | `applied` | ATS confirmation email |
| `interview_invite` | `phone_screen` | Any interview or screen invitation |
| `rejection` | `rejected` | Valid from: `applied`, `phone_screen`, `interview_loop`, `offer` |
| `offer` | `offer` | Formal or informal offer |
| `other` | *(none)* | Recruiter outreach, informational — log only, no transition suggestion |

Emails that are not job-related should not produce a signal at all (skip, do not store).

### Pipeline states (lowercase, matching DB)

`discovered` → `shortlisted` → `approved` → `materials_generated` → `applied` → `phone_screen` → `interview_loop` → `offer` → `accepted`

Valid transitions for email signal types:

- `rejected`: reachable from `applied`, `phone_screen`, `interview_loop`, `offer`
- `withdrawn`: reachable from `discovered` through `offer` (any pre-terminal state)
- `archived`: reachable from `accepted`, `rejected`, `withdrawn`

### Confidence threshold

The automated pipeline uses a single cutoff: **confidence >= 0.6** to store a signal. Below 0.6, the signal is discarded.

> **No transition is applied without user confirmation.** `confirm_signal` is always required. The `suggested_transition` in the signal record is a recommendation only — a simple lowercase string matching a pipeline state name (e.g., `"rejected"`, `"phone_screen"`). After confirmation, the backend can apply the transition against a linked or overridden pipeline entry, or create a normalized opportunity when you provide `opportunity_data`.

## Classification

### Mode 1: Product-triggered automation

The product may trigger the same underlying Gmail and signal tools outside this skill. From the skill's perspective, rely only on the MCP contracts documented here rather than on any specific scheduler, agent runtime, or background cadence.

### Mode 2: Manual sync (this skill)

When running a manual sync, you are the classifier. Use the decision tree below.

### False positive prevention (CRITICAL)

**Before classifying any email, check for marketing indicators.** If ANY of these appear in the email, it is NOT job-related — skip it entirely:

- "unsubscribe", "opt out", "view in browser", "email preferences", "manage preferences"
- "% off", "promo code", "limited time", "sale ends", "coupon", "free trial"
- "shop now", "order now", "buy now", "add to cart", "special offer"
- Bulk sender domains: mailchimp, sendgrid, constantcontact, hubspot, marketo

**"Offer" is NOT always a job offer.** "Special offer", "limited offer", "exclusive offer" are marketing. Only classify as `offer` when the context is clearly about employment (e.g., "We are pleased to extend an offer for the Senior Engineer position").

**Single generic words are not signals.** Words like "opportunity", "position", "role", "interested", "call", "meet" appear in marketing emails constantly. Only classify based on these words when the full context is clearly about a specific job application.

**Pipeline context check.** Use the existing pipeline only as review context while classifying likely company relevance; do not imply that sender-domain heuristics create or attach pipeline links. If the sender has no clear pipeline context and the email has marketing indicators, skip it.

### Classification decision tree

1. **FIRST: Check for marketing indicators** (see above)
   → If found: skip — do not store a signal

2. **Check subject + body for rejection signals**: "unfortunately", "not moving forward", "decided to pursue other candidates", "position has been filled"
   → `signal_type: "rejection"`, confidence: 0.85–0.95

3. **Check for interview/screen signals**: "phone interview", "interview invitation", "schedule an interview", "phone screen", "next round"
   → `signal_type: "interview_invite"`, confidence: 0.70–0.90

4. **Check for offer signals**: "job offer", "offer letter", "pleased to extend an offer", "compensation package"
   → `signal_type: "offer"`, confidence: 0.80–0.95

5. **Check for application confirmation**: "received your application", "application confirmed", "thank you for applying"
   → `signal_type: "confirmation"`, confidence: 0.70–0.85

6. **Check for recruiter outreach about a SPECIFIC role** (not generic spam): named company + named role + personalized message
   → `signal_type: "other"`, confidence: 0.55–0.70

7. **No match, generic, or ambiguous**
   → skip — do not store a signal

### Confidence adjustments

- Exact subject keyword match: use the higher end of the range
- Body-only signal (no subject match): use the lower end
- Sender domain matches a company in the user's pipeline: +0.05
- Generic sender (noreply@, donotreply@): −0.05

> **Decision Point — ATS template emails**: Modern ATS platforms (Greenhouse, Lever, Workday) often use generic subjects like "Update on your application." When the subject is generic but the body clearly contains a signal, classify based on body content and use mid-range confidence.
