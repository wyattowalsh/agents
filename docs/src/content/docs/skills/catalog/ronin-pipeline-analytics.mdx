## What It Does

Analyze job search pipeline performance — funnel conversion rates, stage velocity, rejection patterns, goal tracking, and market benchmarking. This skill reads pipeline data and computes derived metrics. It never transitions pipeline state, modifies preferences, or creates/deletes entities.

## Overview

This skill provides a **read-only analytics layer** over the Ronin pipeline. It consumes data produced by other skills (discovery, research, materials, email-monitor) and derives performance insights that inform the user's job search strategy.

### Read-Only Principle

This skill **never calls** any of the following tools:

- `transition_pipeline` — no state changes
- `create_opportunity` / `approve_opportunity` / `reject_opportunity` — no entity creation or modification
- `update_preferences` — no preference changes
- `create_research` / `create_materials` — no content generation

Every tool in the `allowed-tools` list is a read operation or an external search. If the analysis reveals actionable changes (e.g., "your discovery rate is too low"), recommend the appropriate skill to invoke — do not take the action directly.

## Pipeline States and Funnel Mapping

All states are **lowercase** (matching the DB constraint). The `get_funnel_stats` tool maps 12 pipeline states into 5 funnel buckets:

| Pipeline state | Funnel bucket | Description |
|---------------|--------------|-------------|
| `discovered`, `shortlisted` | discovered | Found / user-approved for review |
| `approved`, `materials_generated` | approved | Cleared for materials / materials ready |
| `applied` | applied | Application submitted |
| `phone_screen`, `interview_loop` | interview | Phone screen / active interview rounds |
| `offer`, `accepted` | offer | Offer received / accepted |
| `rejected`, `withdrawn`, `archived` | (terminal) | Excluded from funnel; tracked via `terminal_count` |

Funnel counts are **cumulative from the bottom up**: `discovered_count` includes everything that reached at least the discovered stage. `approved_count` includes everything that reached approved or deeper. And so on down to `offer_count`.

## Workflow 1: Funnel Analysis

Analyze conversion rates across the pipeline funnel to identify bottleneck stages.

### Step 1 — Load funnel stats

```python
mcp__ronin__get_funnel_stats(user_id="<uuid>")
```

### Step 2 — Identify bottlenecks

| Transition | Healthy range | Bottleneck signal |
|-----------|--------------|-------------------|
| discovered → approved | 30-60% | < 30% means discovery is too broad or preferences are misaligned |
| approved → applied | 60-90% | < 60% means materials generation is stalling or user is hesitating |
| applied → interview | 20-40% | < 20% means applications are not landing — review materials quality |
| interview → offer | 25-50% | < 25% means interview performance needs work — run `ronin-interview-prep` |

> These ranges are heuristics, not absolutes. A user applying only to top-tier companies will have lower applied → interview rates.

### Step 3 — Present the funnel

Present as a visual funnel with conversion percentages between stages. Flag the strongest drop-off point. Recommend specific actions based on the bottleneck (e.g., "Your biggest drop-off is applied → interview. Consider running `ronin-interview-prep` on your active applications.").

## Workflow 2: Velocity Metrics

Compute how fast entries move through the pipeline — average time-in-state per stage and week-over-week velocity trends.

### Step 1 — Load all pipeline entries

```python
mcp__ronin__get_pipeline(user_id="<uuid>")
```

### Step 2 — Load transition history for each entry

```python
mcp__ronin__get_pipeline_history(user_id="<uuid>", entry_id="entry-uuid-1")
```

Transitions are in **ascending chronological order** (`created_at` ASC). The first transition has `from_state: null` (initial creation).

> **Batch strategy:** Issue `get_pipeline_history` calls in parallel for all entries. If the pipeline has >20 entries, prioritize active states (applied, phone_screen, interview_loop, offer) and sample from earlier stages.

### Step 3 — Compute time-in-state

For each entry with transition history, compute duration between consecutive transitions:

```
time_in_state(stage) = transition[n+1].created_at - transition[n].created_at
```

where `transition[n].to_state == stage`.

For entries still in their current state (no subsequent transition):

```
time_in_current_state = now - last_transition.created_at
```

### Step 4 — Aggregate velocity metrics

```
avg_time_in_state(stage) = sum(time_in_state(stage) for all entries) / count(entries that passed through stage)
```

Present as a table showing stage, average time-in-state, entries measured, and interpretation.

### Step 5 — Week-over-week velocity trends

Group transitions by calendar week (ISO week number). For each week, count:

- **Entries created** — transitions with `from_state: null`
- **Entries advanced** — transitions to a deeper funnel stage
- **Entries terminated** — transitions to rejected, withdrawn, or archived

Flag declining trends: "Discovery velocity dropped from 8/week to 3/week. Consider running `ronin-discovery` to refill the pipeline."

## Workflow 3: Rejection Pattern Analysis

Analyze rejected and withdrawn entries to identify patterns — which stages have the highest rejection rates, and whether rejections correlate with opportunity attributes.

### Step 1 — Load terminal entries

```python
rejected = mcp__ronin__get_pipeline(user_id="<uuid>", state="rejected")
withdrawn = mcp__ronin__get_pipeline(user_id="<uuid>", state="withdrawn")
```

### Step 2 — Load transition history for terminal entries

For each rejected/withdrawn entry, fetch the transition history to determine the **stage at rejection**. The `from_state` on the last transition tells you where the entry was when it was rejected/withdrawn.

### Step 3 — Load opportunity details for correlation

```python
mcp__ronin__get_opportunity(user_id="<uuid>", opportunity_id="<opp-id>")
```

### Step 4 — Categorize rejection patterns

**By stage at rejection** — count and percentage of rejections at each stage.

**By company attribute:**
- Companies where multiple rejections occurred (repeat rejection targets)
- Average match_score of rejected entries vs. active entries
- Required skills that appear in rejected entries but not the user's profile (skill gap signal)

**By timing:**
- Entries rejected within 24h of creation (snap rejections — likely low-quality matches)
- Entries rejected after long dwell time (> 2 weeks — stale pipeline items)

### Step 5 — Present actionable patterns

Present the top patterns with specific recommendations (e.g., "75% of rejections happen at discovered or applied stage — discovery quality may need tuning", or "4 of 5 post-applied rejections had 'Java' in required_skills which is not in your profile").

## Workflow 4: Goal Tracking

Compare current pipeline state and velocity against the user's stated goals. Project whether the user is on track, and compute what inputs are needed to hit the target.

### Step 1 — Load context (parallel)

```python
funnel = mcp__ronin__get_funnel_stats(user_id="<uuid>")
prefs = mcp__ronin__get_preferences(user_id="<uuid>")
memory = mcp__ronin__get_user_memory(user_id="<uuid>")
pipeline = mcp__ronin__get_pipeline(user_id="<uuid>")
```

Extract: `compensation_goals`, `target_roles` from preferences. Goals and timeline constraints from memory (e.g., "I want 3 offers in 8 weeks").

### Step 2 — Compute end-to-end conversion rate

```
discovered_to_offer_rate = offer_count / discovered_count
```

If `discovered_count` is 0, there is no data to project from — recommend running `ronin-discovery`.

### Step 3 — Project pipeline needs (inverse funnel formula)

```
needed_discovered = target_offers / (offer_count / discovered_count)
needed_approved   = target_offers / (offer_count / approved_count)
needed_applied    = target_offers / (offer_count / applied_count)
needed_interview  = target_offers / (offer_count / interview_count)
```

If any stage count is 0, the rate is undefined — use industry heuristics (see Workflow 5) as fallback.

### Step 4 — Compute timeline projection

Using velocity data from Workflow 2 (or estimate from available transition timestamps):

```
avg_days_discovered_to_offer = sum of avg_time_in_state for all intermediate stages
weeks_to_target = (needed_discovered - discovered_count) / weekly_discovery_rate + avg_days_discovered_to_offer / 7
```

### Step 5 — Present goal tracking dashboard

Present: current pipeline counts, end-to-end rate, number of entries needed, weekly discovery rate, time projection, on-track/off-track status, and recommended actions.

If the user has not stated explicit goals, prompt: "What is your target? For example: 'I want 2 offers within 6 weeks' or 'I need to accept a role by July 1'."

## Workflow 5: Market Benchmarking

Contextualize the user's pipeline performance against published job search statistics.

### Step 1 — Gather benchmark data

Use the published benchmark ranges below as starting points. For more current data, recommend `ronin-search-strategy` which has web search tools to find updated market statistics.

### Step 2 — Extract published benchmarks

Typical published benchmarks (use as starting points — always cite actual search results):

| Metric | Industry average | Source context |
|--------|-----------------|---------------|
| Applications per offer | 100-200 | General job market |
| Application → interview rate | 5-15% | Varies by seniority and market |
| Interview → offer rate | 20-40% | Depends on company selectivity |
| Average job search duration | 3-6 months | Bureau of Labor Statistics |
| Response rate (applied → any response) | 10-25% | LinkedIn / Glassdoor surveys |

### Step 3 — Compare user performance

Present a table comparing the user's conversion rates, end-to-end rate, and pipeline velocity against market averages.

### Step 4 — Contextualize and caveat

Always include caveats:

- Published benchmarks cover all industries and seniority levels. A staff-level engineer targeting top-tier companies will have different rates.
- Small sample sizes (< 20 entries) make individual rates unreliable. Flag this.
- Market conditions change. 2025-2026 tech hiring may differ from historical averages.
- The user's discovery process is pre-filtered by match scoring, so their application → interview rate should exceed the general average.

## Error Handling

See [`references/error-reference.md`](references/error-reference.md) for the full error table and limitations.

Key errors to watch for:
- `get_funnel_stats` returns all counts as 0 → run `ronin-discovery` first
- `get_pipeline_history` returns `error: "not_found"` → verify entry_id against `get_pipeline` results
- Velocity metrics produce `NaN` → too few entries with transition history
- MCP connection unavailable → reconnect the Ronin MCP server configured for this harness, or fix the local API-mounted `/mcp` setup if you are explicitly running Ronin locally
